## 1\. Redis 缓存与数据库核心区别

### 1\.1 本质差异

|维度|Redis（缓存）|MySQL/PostgreSQL（磁盘数据库）|
|---|---|---|
|存储介质|内存为主，落盘仅备份|磁盘持久化存储|
|读写性能|微秒级，单机 10w\+ QPS|毫秒级，单机几千 QPS 上限|
|数据特性|热数据、短期缓存、允许短暂不一致|全量业务数据、ACID 强一致、事务|
|生命周期|支持过期淘汰，冷数据自动清理|永久存储，不会自动删除业务数据|
|核心定位|拦截高并发读，减轻数据库压力|可靠存储核心业务数据，保证不丢失|

### 1\.2 缓存读写流程

1. 客户端请求先查询 Redis

    - 命中：直接返回，**完全不访问数据库**（90% 以上流量）

    - 未命中（无缓存 / 过期 / 清空）：查询数据库

2. 将数据库结果写入 Redis，设置过期时间后返回

3. 只有少量冷数据、新数据会穿透到数据库，避免数据库雪崩

### 1\.3 两者不可替代

- Redis 不能替代数据库：内存昂贵、断电易丢、无复杂事务、不支持海量持久存储

- 数据库不能替代缓存：磁盘 IO 慢，无法支撑活动大并发流量

## 2\. Redis 主从复制机制

### 2\.1 核心标识：复制 ID（repl\-id）\+ 复制偏移量（offset）

1. **复制 ID repl\-id**
主节点唯一标识，用于校验从库是否同步过当前主库。ID 不一致直接触发全量同步；同步完成后从库会覆盖自身 ID 与主库保持统一。

2. **复制偏移量 offset**
主库每执行一条写命令 offset \+1；从库记录自身同步到的 offset，用于判断数据差距。

### 2\.2 两种同步模式

#### （1）增量部分同步（CONTINUE，推荐）

触发条件：

- 主从 repl\-id 完全一致

- 从库上报的 offset 仍存在于**复制积压缓冲区**
流程：主节点补发 offset 之后所有缺失命令，从库本地重放对齐，无 RDB 传输，性能损耗极低。
适用场景：从库短暂网络抖动、短时间断线重连。

#### （2）全量同步（FULLRESYNC）

触发条件（满足其一即触发）：

1. 复制 ID 不匹配（新从库、主库重启、从库切换主库）

2. offset 被复制积压缓冲区循环覆盖（从库离线太久）
流程：

3. 主库 `BGSAVE` 后台生成完整 RDB 快照，记录快照截止 offset

4. 快照生成期间新写入命令存入复制积压缓冲区

5. 完整 RDB 文件传输给从库，从库清空本地数据加载快照

6. 补发缓冲区快照期间新增命令，从库重放完成数据对齐
缺点：消耗大量带宽、磁盘 IO，同步期间从库无法提供服务。

### 2\.3 复制积压缓冲区

1. 主节点固定大小环形内存（默认 1MB，参数`repl-backlog-size`可调）

2. 存储近期所有写命令 \+ 对应 offset，仅为断线从库增量同步兜底

3. 环形覆盖机制：离线过久，旧 offset 被新数据擦除，只能走全量同步

### 2\.4 主从基础特性

1. 复制异步：主库写完本地直接响应客户端，异步同步从库，存在短暂数据不一致

2. 从库默认只读，禁止写入避免数据冲突

3. 全量同步使用`BGSAVE`fork 子进程，不阻塞主库业务

## 3\. Redis 哨兵（Sentinel）高可用

### 3\.1 哨兵三大核心职能

1. **监控 Monitor**：持续 PING 所有主 / 从节点，探测存活状态

2. **通知 Notify**：同步集群状态给其他哨兵、推送主库变更给业务客户端、支持运维告警脚本

3. **故障转移 Failover**：主库故障时自动选举最优从库升级为新主，重构复制关系

### 3\.2 故障判定两层标准（所有哨兵规则统一）

1. **主观下线 SDOWN**
单个哨兵连续超时 PING 不通主库，单方面标记故障，仅代表单哨兵视角。

2. **客观下线 ODOWN**
超过配置`quorum`（法定投票数）数量的哨兵同时标记 SDOWN，集群确认主库真实故障，才启动故障转移。

    > 生产规范：哨兵部署奇数台（3/5 个），quorum = 哨兵总数 / 2 \+1，防网络分区误判。
    > 
    > 

### 3\.3 领头哨兵选举机制（无 AI、无权威节点）

1. 集群判定主库 ODOWN 后，所有哨兵竞争竞选领头哨兵

2. 每个哨兵一轮故障转移仅可投一票，投给第一个发来竞选请求的哨兵

3. 第一个拿到**过半哨兵选票**的节点当选领头，全权执行故障切换操作

4. 一轮选举无人过半则随机等待重试，避免多哨兵同时操作引发冲突

### 3\.4 故障转移完整流程

1. 领头哨兵筛选最优从库（优先级从上至下）
① 排除下线、同步中断的不健康从库
② `replica-priority`优先级数值越小越优先（0 代表永不升主）
③ 优先级一致，复制 offset 越大（数据越新）优先
④ offset 相同，选取复制 ID 字典序更小节点

2. 向选中从库发送`replicaof no one`，升级为独立新主库

3. 所有剩余旧从库重置复制，指向新主节点同步数据

4. 旧主库故障恢复上线后，哨兵强制其成为新主的从库，不会自动抢回主身份

### 3\.5 脑裂防护配置

```conf
min-replicas-to-write 1
min-replicas-max-lag 10
```

主库必须至少有 1 台延迟低于 10 秒的在线从库，才接受客户端写请求；网络分区脑裂时拒绝写入，规避数据丢失。

### 3\.6 哨兵模式整体架构

1 套主从（1 主 N 从）\+ 多个独立哨兵进程；**不分片数据**，全量数据存储在一套主从中，仅解决高可用、自动故障切换，无法水平扩容存储。

## 4\. Redis Cluster 分片集群深度解析

### 4\.1 传统哈希分片致命缺陷

公式：`hash(key) % 机器总数`
扩容 / 缩容时机器数量变更，绝大多数 key 映射节点全部失效，需要全量数据迁移，阻塞服务，线上不可用。

### 4\.2 核心设计：16384 固定哈希槽 Hashslot

1. 计算规则：`slot = CRC16(key) % 16384`

2. 16384 为固定常量，任意 key 对应的 slot 永久不变；仅调整槽位归属的主节点，实现平滑扩容。

3. 16384 设计原因：元数据心跳包体积仅 16KB，平衡带宽与扩容迁移粒度。

### 4\.3 槽位扩容迁移流程（示例：2 主扩容至 3 主）

1. 初始状态：M1 \(0\\8191\)、M2 \(8192\\16383\)

2. 新增主节点 M3，集群自动拆分槽位
M1 \(0\\5460\)、M2 \(5461\\10922\)、M3 \(10923\~16383\)

3. 以**哈希槽**为最小迁移单位，批量迁移槽内所有 key

4. 迁移过程不中断集群读写：访问迁移中槽位触发`ASK`临时重定向；迁移完成触发`MOVED`永久重定向更新客户端缓存

### 4\.4 Gossip 集群通信协议

1. 所有主 / 从节点定时互相发送心跳包，同步全局元数据：节点地址、槽位分配、节点健康状态、投票信息

2. 故障判定逻辑复用主从 \+ 投票机制：PFAIL（单节点主观下线）→ 过半主节点确认后标记 FAIL（客观下线）

3. 主节点故障后，自动从自身从库选举 offset 最大的节点升主，接管全部哈希槽

### 4\.5 Cluster 复制架构规范

1. 仅**主节点分配哈希槽**，负责读写；从节点无槽位，仅做数据副本

2. 每个主节点至少配置 1 个从库，主宕机自动切换保证数据可用

3. 不支持多级主从链，仅允许一层主从复制

### 4\.6 Cluster 硬性限制

1. 仅支持 db0，无法使用`select`切换逻辑数据库

2. 多 key 操作（mget/mset/ 事务 / Lua）key 必须落在同一槽；解决方案：哈希标签`{tag}key`，大括号内字符参与槽计算

3. 集群上线前 16384 个哈希槽必须全部分配，缺槽集群无法对外提供读写

4. 下线主节点前必须迁移完其全部槽位，不可直接销毁

### 4\.7 集群脑裂规范

主节点数量必须为**奇数**（3/5/7 主），过半投票才能判定故障，避免网络分区票数均等无法决策。

## 5\. Redis 持久化：RDB / AOF / Fork 写时复制

### 5\.1 三种持久化模式

#### （1）无持久化

完全关闭 RDB、AOF，内存断电数据全部丢失；仅适用于临时纯缓存场景。

#### （2）RDB 快照持久化

- 原理：定时执行`BGSAVE/SAVE`生成二进制完整数据快照文件`.rdb`，记录某一瞬间全量内存数据

- 优点：文件压缩体积小、服务重启加载速度极快

- 缺点：两次快照之间写入的数据会丢失；fork 子进程存在瞬时内存开销

- 生成方式：

    - `SAVE`：主线程阻塞生成，线上禁止使用

    - `BGSAVE`：fork 子进程后台生成，主线程正常处理业务

#### （3）AOF 追加日志持久化

- 原理：逐条记录客户端所有写命令，服务重启时重放命令恢复完整数据

- 三种刷盘策略：

    1. `always`：每条命令立刻落盘，几乎零数据丢失，性能极差

    2. `everysec`：每秒刷盘，最多丢失 1 秒数据，生产主流方案

    3. `no`：交给操作系统自动刷盘，丢失数据不可控

- 优化：`bgrewriteaof`AOF 重写，合并重复命令压缩日志文件

- 优点：数据丢失量极低；缺点：日志文件体积大，重启恢复速度慢

#### （4）混合持久化（生产推荐）

同时开启 RDB\+AOF；重启优先加载 RDB 快速恢复基准数据，再追加 AOF 增量日志，兼顾加载速度与数据安全性。

### 5\.2 Fork \+ 写时复制 Copy\-On\-Write（RDB 底层核心）

Redis 为单线程程序，依靠操作系统 COW 机制实现无阻塞快照：

1. 执行`fork()`创建子进程，瞬间父子进程共享全部内存页，不复制内存，开销极低

2. 子进程只读原始内存，持续生成一致性 RDB 快照

3. 父进程处理新写入：被修改的内存页操作系统单独复制一份新页面（写时复制）

4. 仅修改过的内存页面才复制，极大节省内存资源，不阻塞主线程业务

## 6\. 哨兵模式 VS Redis Cluster 完整对比

|对比维度|Sentinel 哨兵模式|Redis Cluster 分片集群|
|---|---|---|
|数据分片|不分片，一套全量主从|16384 哈希槽分片，多组独立主从|
|扩容能力|无法水平扩容内存，受单台机器内存限制|多机器分摊数据，支持海量缓存水平扩容|
|高可用实现|额外独立哨兵进程监控、故障切换|节点内置 Gossip 协议，无需额外哨兵|
|核心用途|数据量小，仅需自动故障转移高可用|百万 / 亿级 Key，需要分摊内存、大并发存储|
|客户端连接|连接哨兵获取当前主库地址|直连任意集群节点，支持 MOVED/ASK 重定向|
|多库支持|支持`select`切换 db|仅支持 db0|

## 7\. 高频误区与生产规范

### 7\.1 常见误区

1. 缓存有数据库查询逻辑 = 不如直接存库
纠正：80% 流量集中在 20% 热点数据，缓存拦截绝大多数请求，仅少量请求穿透数据库，大幅降低数据库压力。

2. 偏移量差值大就一定会全量同步
纠正：优先判断 repl\-id，ID 不匹配无论 offset 差值多少都全量；ID 匹配但 offset 被缓冲区覆盖才会全量。

3. 领头哨兵依靠性能、权威优先级选举
纠正：纯分布式过半投票，哨兵地位完全平等，与负载、运行时长无关。

4. 集群 = 多个哨兵 / 集群 = 多套主从
纠正：集群是通用分布式名词；哨兵模式是 1 套主从 \+ 多哨兵（不分片），Redis Cluster 是多组主从分片集群。

5. 主库故障恢复后自动变回主节点
纠正：旧主上线后自动成为新主的从库，不会自动抢占主身份。

### 7\.2 线上生产规范

1. 哨兵：奇数台独立部署，quorum 过半配置，开启最小从库写入防脑裂

2. 主从复制：调高`repl-backlog-size`减少全量同步，从库默认只读

3. Redis Cluster：主节点奇数部署，每主至少 1 从，全部开启混合持久化

4. 持久化：禁用`SAVE`阻塞快照，AOF 使用 everysec 刷盘，定期执行 AOF 重写

5. 扩容迁移：在线迁移哈希槽，禁止直接下线持有槽位的主节点

6. 安全：公网不直接暴露 Redis，设置强密码，限制 IP 访问，禁止 root@% 无限制远程账号
